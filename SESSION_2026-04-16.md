## Broken / left for tomorrow

### Voice — Vapi silence timeout

Test call: Anna answered, user said "I'm looking for detailing," Anna said "pulling up schedule" → 2-3s silence → call dropped. Call log shows **`silence-timed-out`**.

Our tool endpoint returns in ~200ms (verified via curl). So the silence is happening **after** the tool result comes back — meaning Claude Haiku isn't producing the next turn fast enough, OR isn't producing it at all.

### Most likely causes (untested)

1. **Tools not actually being called** — Anna's "pulling up schedule" is just filler text she hallucinated. The actual tool call may never have been dispatched if the tools aren't properly attached to the assistant's Model tab (not just existing in the global Tools tab). Vapi separates "tool exists" from "tool is enabled for this assistant."
2. **Haiku prompt issue** — system prompt may not make it clear enough that she MUST call `get_slots` before offering times. She might be trying to answer without the tool.
3. **Vapi silence timeout threshold too short** — default ~10s. If Haiku takes 3-5s after tool returns, it's tight.
4. **Response format mismatch** — Vapi may expect a slightly different response shape than what we return. (Curl test used a synthetic payload, not a real Vapi payload — possible we got the shape *close* but not *exact*.)

### Decision to make tomorrow

- **Option A:** Screenshot Vapi call log Timeline → find exactly where silence begins → surgical fix.
- **Option B:** Ditch Vapi, switch to ElevenLabs Conversational AI. Tool endpoints are provider-agnostic — zero code change, just paste URLs into a new dashboard. ElevenLabs UX is significantly cleaner. Probably the right move given time pressure.

Recommendation: **Option B**. We've already sunk enough into Vapi debugging. ElevenLabs ConvAI setup is ~30 min from scratch and we keep 100% of the backend work.
